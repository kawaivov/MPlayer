## 1. Мета роботи
1. Ознайомитись із базовою архітектурою веб-застосунку (MVC/MTV).
2. Реалізувати бекенд на Python (Flask) та налаштувати взаємодію з базою даних.
3. Створити клієнтську частину для відображення та керування даними.
4. Освоїти навички роботи з системою контролю версій Git (репозиторій, коміти, гілкування).

## 2. Опис проєкту
В якості теми командного проєкту обрано **Музичну стрімінгову платформу**. Це веб-застосунок, що дозволяє користувачам завантажувати аудіофайли, редагувати їх метадані, прослуховувати музику через вбудований плеєр та керувати своєю бібліотекою.

**Основний функціонал:**
* Реєстрація та авторизація користувачів.
* Завантаження треків (аудіо + обкладинка).
* Відтворення музики (Custom Audio Player).
* "Живий" пошук по базі даних.
* Редагування та видалення треків.

## 3. Технічна реалізація

### 3.1. Стек технологій
* **Бекенд:** Python 3, Flask.
* **База даних:** SQLite3.
* **Фронтенд:** HTML5, CSS3 (Custom styles), JavaScript (Vanilla).
* **Шаблонізатор:** Jinja2 (використано підхід Server-Side Rendering).

### 3.2. База даних
Було спроєктовано структуру бази даних у файлі `music_platform.db`.
**Таблиця `users`:**
* `id` (PK)
* `username`
* `email`
* `password_hash` (паролі зберігаються у зашифрованому вигляді)

**Таблиця `songs`:**
* `id` (PK)
* `title` (Назва пісні)
* `artist` (Виконавець)
* `uploader` (Нікнейм користувача, який завантажив)
* `filename` (Ім'я файлу на сервері)
* `cover_filename` (Ім'я файлу обкладинки)
* `is_visible` (Статус: активна/прихована)

### 3.3. Бекенд (Flask)
Реалізовано REST-подібну архітектуру з наступними маршрутами:

| Метод | Маршрут | Опис |
|-------|---------|------|
| GET | `/` | Головна сторінка (відображення списку пісень) |
| POST | `/register` | Реєстрація нового користувача |
| POST | `/login` | Вхід у систему (створення сесії) |
| GET | `/logout` | Вихід із системи |
| POST | `/upload` | Завантаження файлів на сервер |
| POST | `/edit/<id>` | Оновлення даних пісні або обкладинки |
| POST | `/delete/<id>` | Видалення запису та файлів |

### 3.4. Клієнтська частина та Інтерфейс
* **Дизайн:** Розроблено темну тему в стилі Spotify. Використано Flexbox для адаптивності.
* **Аудіоплеєр:** Створено кастомний плеєр (HTML+JS), який замінює стандартний тег `<audio>`. Реалізовано смугу прокрутки (seek bar) з градієнтним заповненням.
* **Пошук:** Реалізовано фільтрацію DOM-елементів на стороні клієнта для миттєвого пошуку без перезавантаження сторінки.

## 4. Робота з Git
У ході роботи було використано систему контролю версій Git.
1. Ініціалізовано локальний репозиторій.
2. Налаштовано файл `.gitignore` для ігнорування віртуального середовища (`venv/`) та скомпільованих файлів.
3. Робота велася за принципом: створення гілки -> внесення змін -> коміт -> злиття з `main`.

Приклад команд, що використовувались:
```bash
git init
git add .
git commit -m "Implement authentication and CRUD"
git push origin main
```


## CI/CD Implementation (Practice 7)

**Реалізовано:**
1.  **CI:** Автоматичний запуск тестів (`unittest`) при кожному пуші та PR.
2.  **CD:** Автоматична збірка та публікація Docker-образів у **GitHub Container Registry**.
    * Image API: `ghcr.io/kawaivov/mplayer-api`
    * Image Web: `ghcr.io/kawaivov/mplayer-web`
3.  **Style Points:**
    * ✅ **Concurrency:** Скасування застарілих білдів для економії ресурсів.
    * ✅ **Permissions:** Використано принцип найменших привілеїв (`packages: write` тільки для job публікації).
    * ✅ **Semantic Tagging:** Образи тегуються як `latest` та `sha-<commit_hash>`.
    * ✅ **Matrix Strategy:** Оптимізована збірка кількох сервісів (API + Web) в одному workflow.
